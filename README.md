# GovAI Assistant (Norway) — PathBoot2

A privacy-first, local AI-powered government guidance system to help citizens — especially immigrants in Norway — navigate public services such as **NAV**, **tax**, and **immigration**. Users ask questions in natural language and receive **clear, structured, multilingual** answers generated by a **local LLM** with **retrieval-augmented generation (RAG)**.

> **Example:** "How do I apply for unemployment benefits in Norway?"
> The assistant returns structured answers and supports at least **English + Norwegian + Amharic**.

## Project Overview

### Purpose & Impact

GovAI Assistant improves **digital inclusion and accessibility** by reducing the friction of navigating complex government websites and procedures — especially for newcomers facing language barriers.

**Key benefits**
- Makes Norwegian public services easier to understand (NAV, Tax, Immigration)
- Multilingual support (target: **English, Norwegian, Amharic**)
- Step-by-step guidance that is easier to follow than dense web pages
- Reduced dependency on support centres and long wait times
- Privacy-first approach using **local inference** wherever possible

### What this repository contains

This repo is the **backend-first** implementation (Spring Boot + AI orchestration) designed to integrate with a modern chat UI (recommended stack described in [Frontend](#frontend)).

## Architecture (High-Level)

**Request flow**

```
User question (UI / API)
        │
        ▼
Session memory (stores conversation context)
        │
        ▼
Language detection / normalisation
        │
        ▼
Domain router  ──► NAV (welfare) | Tax | Immigration
        │
        ▼
Agent factory  ──► Domain-specialised agent
        │
        ▼
RAG pipeline  ──► Vector embeddings + similarity search
        │
        ▼
Prompt composition (context + history + instructions)
        │
        ▼
Local LLM (Mistral via Ollama)
        │
        ▼
Translation layer (back to user language)
        │
        ▼
Structured response  ──► stored + returned
```

## MVP Definition

### MVP Goal

Deliver a working end-to-end assistant that:
- Answers questions for at least **one domain** (NAV recommended first)
- Uses **RAG** with a small, curated dataset
- Runs **locally** (LLM + embeddings via Ollama)
- **Returns structured answers and supports at least English + Norwegian + Amharic**

### MVP In-scope (recommended)

| Feature | Priority |
|---|---|
| Single chat endpoint with sessions | **Must have** |
| Domain routing (NAV / Tax / Immigration) | **Must have** |
| RAG retrieval + prompt composition | **Must have** |
| Local inference via Ollama (Mistral) | **Must have** |
| Basic translation layer or bilingual prompts | **Must have** |
| Health endpoint + basic logging | **Must have** |
| Minimal unit tests (router + prompt builder) | **Must have** |

### Out of scope for MVP

- Production-grade auth & billing
- Large-scale document ingestion pipelines
- Full UI polish / mobile app
- Complex observability & multi-region infrastructure

## Backend

### Core Components

- **Spring Boot REST API** — entry point for all client requests
- **Orchestrator service** — pipeline coordination
- **Domain router** — classifies questions into NAV / Tax / Immigration
- **Agent factory** — creates domain-specialised agents with targeted prompt strategies
- **RAG pipeline**
  - Embeddings (e.g., `nomic-embed-text` via Ollama)
  - Vector store (implementation-defined)
- **Local LLM integration** — via Ollama (Mistral recommended)
- **Translation layer** — language detection and response translation
- **Session memory store** — per-user conversation history

### Suggested API Endpoints

#### Must have

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/chat` | Send a message and receive a structured answer |
| `GET` | `/api/health` | Liveness / readiness probe |

#### Should have

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/sessions` | Create a new session |
| `GET` | `/api/sessions/{id}` | Fetch session history |
| `POST` | `/api/admin/ingest` | Ingest or update documents (secured) |

#### Nice to have

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/metrics` | Prometheus-compatible metrics |
| `POST` | `/api/feedback` | Capture user feedback (thumbs up/down + comment) |

## Setup & Run (Local)

### Prerequisites

- Java 17+
- Maven
- Git
- [Ollama](https://ollama.com/) installed and running

### 1 — Pull LLM and embedding models (Ollama)

```bash
ollama pull mistral
ollama pull nomic-embed-text
ollama serve
```

### 2 — Run the backend

```bash
./mvnw spring-boot:run
```

Backend default: `http://localhost:8080`

### 3 — Run the frontend (if present)

```bash
cd frontend
npm install && npm run dev
```

Frontend default: `http://localhost:3000`

## Final Recommendation

1. **Ship a NAV-focused MVP first** with a small, high-quality dataset and a strong answer structure (steps, requirements, links, next actions).
2. **Implement RAG + citations early** — trust and transparency are critical when delivering government guidance.
3. **Keep inference local by default** (Ollama) and explicitly document privacy and data-retention assumptions.
4. **Use Copilot to accelerate backlog and documentation work**, but treat it as an assistant — never as a source of truth for legal or government information.
5. **After MVP**, prioritise streaming UX, ingestion workflows, and automated tests to support safe, incremental iteration.
